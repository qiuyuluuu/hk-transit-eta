# 决策记录

本文记录已经确认过的产品和技术决定，避免后续反复讨论。

## 2026-06-29：第一版使用纯静态网页

决定：先做 `index.html + app.js + styles.css` 的纯前端页面。

原因：

- 当前只查询少量站点/路线
- 九巴 API 支持浏览器跨域访问
- iPhone 可以通过 Safari 添加到主屏幕
- 不需要为第一版引入后端、部署密钥或数据库

后续如果加入通知、缓存或多营运商复杂聚合，再考虑 Cloudflare Worker。

## 2026-06-29：界面使用简体中文

决定：用户界面显示简体中文。

说明：

- 官方 API 同时提供繁体、简体、英文
- 本地路线资料可以保留官方繁体站名
- 面向用户的页面使用简体，例如“逸珑湾”“往大学站方向”

## 2026-06-29：272A 逸珑湾显示为“往大学站方向”

决定：页面显示方向为“往大学站方向”。

原因：

- 九巴官方目的地对循环线常显示为“白石角(循环线)”
- 用户在逸珑湾站点查看时，关心的是车是否快到并继续往大学站方向走
- 生活化方向名比官方循环线目的地更可读

## 2026-06-30：站点标题显示站点编码

决定：站点名称旁直接显示九巴站点编码，例如 `逸珑湾 PA150`、`云汇 PA206`。

原因：

- 272A 存在同名或近似站点，例如 `云滙 (PA125)` 和 `云滙 (PA206)`
- 编码可以帮助确认用户看到的是哪一个物理站点
- 后续新增站点时，编码比纯站名更适合作为界面和文档的稳定标识

## 2026-06-30：云汇默认加入 PA206

决定：本轮加入的是 `云汇 PA206`，方向同样显示为“往大学站方向”。

说明：

- 272A 主循环线中也有 `云滙 PA125`
- `PA206` 是经过逸珑湾和博研路之后、继续往大学站方向的云汇站点
- 如果需要查看另一侧的 `PA125`，后续作为独立站点加入

## 2026-06-30：页面按通勤场景分组

决定：页面分成两个通勤场景：

- `上班`：`往大学、沙田方向`
- `下班`：`往白石角方向`

原因：

- 用户实际使用时更关心“现在是在离开白石角，还是回到白石角”
- 272A 是循环线，官方目的地字段不总是贴合用户语境
- 分组比按路线机械排列更适合手机上快速扫一眼

## 2026-06-30：加入大学站 ST905 回白石角

决定：在下班分组加入 `272A / 大学站 ST905 / 往白石角方向`。

查询 `service_type=1,2,4`。不默认查询 `service_type=3`，因为 `3` 是清晨白石角往大学站的服务，不是大学站开出。

## 2026-06-30：27A 站名用九巴标签、内部保留小巴站点

决定：27A 页面显示沿用九巴站点名和编码，例如 `逸珑湾 PA150`、`云汇 PA206`，但内部使用小巴官方 `route_id / route_seq / stop_seq` 查询 ETA。

当前只加入 27A 普通走线 `route_id=2007861` 的往沙田方向：

- `逸珑湾 PA150` → 小巴 `科进路停车湾近DC2431灯柱`，`stop_seq=4`
- `云汇 PA206` → 小巴 `创新路, 近云滙`，`stop_seq=5`

回白石角方向也使用同一普通走线：

- `沙田站 排头街` → 小巴 `沙田中, 排头街, 港铁沙田站外`，`route_seq=2`，`stop_seq=1`

原因：

- 小巴系统站名多使用道路、屋苑、灯柱描述，和九巴站牌名不同
- 用户实际识别位置时更熟悉当前页面已使用的九巴站名
- route_id 变体很多，先接入普通走线，特殊走线后续单独确认

## 2026-06-30：小巴 ETA 通过同源代理访问

决定：绿色专线小巴 ETA 不由浏览器直接请求，而是通过 Cloudflare Pages Function 暴露同源 `/api/gmb/eta/route-stop/...`。

原因：

- 小巴 API 当前响应头没有 `Access-Control-Allow-Origin`
- 浏览器会因 CORS 拦截直连请求，页面显示“更新失败”
- 服务器端代理可以正常请求小巴 API，再把 JSON 返回给页面

## 2026-06-30：暂不做完整 27A

决定：27A 先只接入普通走线 `route_id=2007861`，暂不把 `2007862 / 2007893 / 2007894` 合并进页面。

暂缓的变体：

- `2007862`：辅助服务，经科技大道西
- `2007893`：特别班次，经海日湾
- `2007894`：特别班次，经海日湾及创新路

原因：

- 这些变体的走线和站序会变化，不是简单的同站不同班次
- `逸珑湾 / 云汇` 在不同变体里可能对应不同小巴官方站点
- 直接合并可能显示并非同一等车位置的 ETA，误导现场判断
- 当前优先目标是把普通走线和小巴代理在 iPhone 上跑稳

触发后续补全的条件：

- 部署试用后发现页面无车但现场有 27A
- 某些时段经常缺少 27A ETA
- 用户明确需要经科技大道西、经海日湾或经海日湾及创新路的班次

## 2026-06-29：272A 查询 service_type 1, 2, 3, 4 并去重

决定：已实现的 `272A` 站点按物理站点和方向精确配置 `service_type`，再按 ETA 时间窗口去重。

当前配置：

- `逸珑湾 PA150`：`1,2,3,4`
- `云汇 PA206`：`1,3,4`
- `大学站 ST905`：`1,2,4`

原因：

- 四种 service type 都经过 `逸瓏灣 (PA150)`
- `service_type=2` 是大学站至博研路短程，不回大学站，因此不适合 `云汇 PA206` 往大学站方向
- `service_type=3` 是清晨白石角往大学站，因此不适合大学站开出的回白石角方向
- 官方可能对同一班车在多个 service type 返回相同 ETA
- 同一 ETA 可能带不同目的地字段，例如“大学站”“博研路”“白石角(循环线)”
- 因此去重不依赖目的地字段，只按同一路线和接近的 ETA 时间判断

当前时间窗口：90 秒。

## 2026-06-29：备注为空显示“实时预计”

决定：当九巴 `rmk_sc` / `rmk_tc` 为空时，页面显示“实时预计”；当备注不为空时显示官方备注。

原因：

- 九巴会返回“原定班次”等备注
- 用户需要区分更可信的实时 ETA 和只按时刻表推算的班次
- 页面不自行计算巴士位置，只展示官方 ETA 与备注语义
